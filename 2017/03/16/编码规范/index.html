Java编码规范

Java
  • 总结
  • 编码规范

前言

本文参考了《代码整洁之道》与《阿里巴巴Java开发手册》，总结一些Java的编程规范，目录参考《阿里巴巴Java开发手册》！

编程规约

命名规约

  • 包名：开头一般以com/cn.组织机构(alibaba)/个人(hc)，所有模块应放在此包名下，新建包名为名词全小写且以模块划分，如：com.hc，新建包名system。
  • 类名：类名和对象名为名词或名词短语，遵从UpperCamelCase风格，如Customer，CustomerController。领域模型(DO、BO、DTO、VO、QO等)命名规约为UpperCamelCase风格 + 领域模型，如CustomerVO。
  • 变量名：名词或名词短语，遵从lowerCamelCase风格，如customer，customerService。
  • 常量名：全大写，单个英语单词下划线隔开，如MAX_SOCKET_COUNT。
  • 方法名：动词或动词短语，遵从lowerCamelCase风格，如save，postPayment。
  • 接口名：定义接口以I+xx，实现接口xx+Impl，如ICustomerService，CustomerServiceImpl。

    函数规约

  • 尽量遵循单一职责。
  • 返回值为boolean类型，以is开头。
  • 函数尽量短小，20行封顶最佳。
  • 尽量使用一元函数。
  • 尽量避免三参数或者三个以上的参数。
  • 出现三参数以上情况，可以考虑封装为类。
  • 每个函数应该只有一个入口和出口。
  • 每个函数只该有一个return函数。
  • 尽量编写少的工具类。
  • 尽量使用架构自带的API。

    注释规约

  • 尽量英文注释。
  • 精炼易懂的语句。
  • 每个代码块入口的参数，需要给出详细注释。
  • 对于可能出现的异常，给出@thrwos注释。
  • 对于过时的方法，给出@deprecated。

    可读性

  • 自顶向下读代码：向下规则。
  • 长而具有描述性的名称比短而令人费解的名称好。
  • 使用异常替代返回错误码，新的异常可以继承Error枚举类。
  • 抽离try/catch代码块，避免错误处理与正常流程混为一谈。
  • 错误处理只做一件事。
  • 循环中尽量避免break、continue。
  • 通过所有测试。
  • 消除重复和提高表达能力，提早构建简单对象。
  • 字母’I’和’O’尽量避免与数字混用。

    单元测试

  • 每个函数模块尽量给出单元测试用例。
  • 对于整个流程给出单元测试用例。
  • 验收测试需要测试每个单元测试，站在客户角度考虑可能出现的情况的整套流程测试。

    结语
